# Hotmail007 独立版安装与配置指南

这份文档面向“直接使用插件的人”，不讲迁移历史，重点只放在：

1. 怎么把扩展装起来
2. 第一次应该配哪些关键项
3. 邮箱模式 / 手机号模式分别怎么跑
4. 常见容易踩坑的地方

## 1. 安装扩展

### 方式 A：加载本地源码目录

1. 打开 `chrome://extensions/`
2. 开启“开发者模式”
3. 点击“加载已解压的扩展程序”
4. 选择目录：

```txt
E:\ECS\C202604291913907\codex-oauth-automation-extension-hotmail007
```

5. 打开扩展侧边栏

### 方式 B：使用独立仓库源码

如果你是从 GitHub 拉源码，优先使用独立仓库：

- [`pkeran/codex-oauth-automation-extension-hotmail007`](https://github.com/pkeran/codex-oauth-automation-extension-hotmail007)

不要再从旧 fork 或上游仓库直接拿 README 作为当前改版插件说明。

## 2. 第一次使用前准备

至少准备三类东西：

### 2.1 OAuth 来源

你需要至少一种可用来源，例如：

- SUB2API
- Codex2API
- 你自己当前已经适配的本地来源

如果这里没配通，后面邮箱和接码都没有意义。

### 2.2 邮箱链路

至少准备一种可用邮箱方案：

- Hotmail007 / Hotmail 账号池
- LuckMail / 2925 / iCloud
- 自定义邮箱池

### 2.3 手机接码链路

如果你会进入手机号验证，还需要至少一种 provider：

- HeroSMS
- 5sim
- NexSMS

并提前确认：

- API key 正确
- 国家列表已配置
- 价格上限不是明显过低
- provider 顺序符合你的实际策略

## 3. 推荐配置顺序

推荐按这个顺序配，而不是一上来把所有开关都开满：

### Step 1：先导入配置

如果你有历史 JSON 配置：

1. 打开侧边栏
2. 进入配置导入入口
3. 选择配置文件
4. 导入后检查关键字段是否真的落进去了

重点检查：

- OAuth 来源配置
- Hotmail007 / Hotmail 账号池
- 接码 provider key
- 自动运行策略
- 记账与账本开关

### Step 2：先确认邮箱能用

优先做最小验证：

- Hotmail007 能采购 / 校验
- Hotmail 账号池能收信
- 自定义邮箱池内容正确

如果邮箱都还没确认可用，不建议直接开 Auto。

### Step 3：再确认手机接码链路

先看三件事：

1. provider 顺序
2. 国家顺序
3. 单价上限

再看恢复策略：

- add-phone 拒绝后是换号、换国家还是换 provider
- 页面限流后是否冷却
- 是否启用自动白嫖复用 / 手动复用

### Step 4：最后再开自动运行

推荐先手动跑一轮关键步骤，再决定是否开多轮：

1. 先确认注册入口正常
2. 再确认验证码链路正常
3. 再确认 OAuth 回调链路正常
4. 最后再开 Auto / never-stop

## 4. 邮箱模式怎么用

邮箱模式下，重点看邮箱来源与验证码链路。

建议优先确认：

- 当前注册邮箱是谁分配的
- 第 4 步验证码由哪个邮箱来源收
- 第 8 步登录验证码由哪个邮箱来源收

如果你使用：

### Hotmail007 / Hotmail

重点检查：

- 账号池里是否真有可用账号
- 校验是否通过
- 本地 helper / 远程服务地址是否正确

### 自定义邮箱池

重点检查：

- 每行一个邮箱
- 当前邮箱池中的邮箱是否和实际收码链路匹配
- Auto 轮次是否按邮箱池数量分配

## 5. 手机号模式怎么用

手机号模式下，重点不是“能不能拿到号”，而是“拿到号后怎么恢复”。

建议优先看这些配置：

### 5.1 Provider 顺序

确保你真的把想要接手的 provider 放进顺序里。

例如只配置 HeroSMS，那么 HeroSMS 无号时不会自动切到 5sim。

### 5.2 国家顺序

建议先放你当前成功率更高、单价也能接受的国家。

### 5.3 恢复策略

至少要明确：

- 无法发码时怎么处理
- 页面限流时怎么处理
- 号码失效时怎么处理
- 订单是否取消
- 是否重开当前轮

### 5.4 白嫖复用

如果你启用了手动复用 / 自动白嫖复用：

- 要确认当前号码确实还可继续使用
- 不要让“复用状态”与“新购号码状态”混在一起

## 6. 自动运行建议

### 推荐起步方式

第一次不要直接跑大批量。

推荐：

1. 单轮手动验证
2. 小批量 Auto
3. 再逐步扩大轮数

### never-stop 模式

如果你启用了 never-stop：

- 插件会尽量在可恢复错误下继续跑
- 但卡在某个阶段超时过久时，仍会按已有策略回到当前轮第一步

所以 never-stop 不是“无脑一直点”，而是“在已定义恢复语义下尽量不中断”。

## 7. 记账与记录建议

如果你在意成本统计，建议同时开启并核对：

- 账号记录
- 成本账本
- 按天账本持久化

你应该重点确认：

1. 邮箱成本是否被记录
2. 手机接码实际消耗是否被记录
3. 失败轮次的已消耗成本是否进入总账本

## 8. 常见问题

### 8.1 为什么我明明导入了配置，还是像没生效？

优先检查：

- 是否导入到了当前正在运行的浏览器 profile
- 是否导入后又被别的旧配置覆盖
- 是否在另一个浏览器 / 另一个 profile 里打开了不同的扩展实例

### 8.2 为什么 provider 有 key，但还是一直无号？

通常先排查：

- 国家顺序是否过窄
- 价格上限是否太低
- provider 顺序是否过严
- 是否只配了一个 provider

### 8.3 为什么记账和成功数量看起来对不上？

因为记账按“实际消耗”统计，不是只按“最终注册成功”统计。

如果某一轮已经消耗邮箱成本或手机接码成本，即使后续失败，也可能进入账本。

## 9. 遇到问题先看哪里

建议按这个顺序看：

1. 侧边栏日志
2. 当前步骤的页面状态
3. 账号记录 / 成本账本
4. 是否命中了已知 structured code
5. 当前 provider / 国家 / 订单状态
